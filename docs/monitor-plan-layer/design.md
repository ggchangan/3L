# 今日计划分组重构 — 设计文档 v1

## 1. 背景与问题

### 现状

「今日计划」(PlanLayer) 是目前展示三类数据：

```
🟢 买入：
  华工科技(000988) 缩量回踩  止损68.5  🔔
  飞沃科技(301232) 区间底部  止损128.88 🔔
🔴 卖出：
  (空)
👁️ 观察：
  德龙激光(688170) → 等待信号确认
───分割线───
🔴 持仓止损：
  华工科技(000988) 止损68.5(-5.2%)  现价71.2 +1.3%
  飞沃科技(301232) 止损128.88(-7.66%)  现价139.5 -1.2%
───分割线───
🟡 计划报警：
  ...
```

### 痛点

1. **信息重复** — 持有华工科技，它在「买入」和「持仓止损」各出现一次，用户要看两遍
2. **分不清归属** — 「买入」里的股票分不清哪些是已有持仓（需要盯止损）、哪些是候选（看买点是否触发）
3. **信息过载** — 三个区块 + 报警，同一只股票的信息被切碎了

### 目标

按「持仓 vs 候选」分组，去重，每个股票只出现一次，信息合并展示。

## 2. 设计思想

### 2.1 核心理念

**持仓优先。** 用户盘中第一要务是看自己的持仓股怎么样了——止损到了没、计划执行到哪了。候选股（非持仓的买点/观察）是次要信息，放在后面。

### 2.2 方案选择

| 候选方案 | 优点 | 缺点 | 结论 |
|:--------|:----|:----|:----:|
| **A: 持仓+候选两分组** | 去重、信息合并、一目了然 | 改动适中 | ✅ |
| B: 只去重不分组（合并显示） | 改动小 | 仍然分不清归属 | ❌ |
| C: 完全平铺带标签 | 不改结构 | 重复还在 | ❌ |

### 2.3 设计原则

- **一个股票只出现一次。** 如果某股同时是持仓和计划项，归入持仓组，合并计划信息。
- **持仓组展示止损+现价。** 候选组只展示条件+止损（不含现价，非持仓不需要实时盯价）。
- **保留可折叠。** 大盘面仍然可折叠节省空间。

### 2.4 屏幕内外

- **做什么：** 今日计划按「持仓操作」+「候选」分组展示，去重合并
- **不做什么：**
  - 不改数据流（计划来源仍是工作台，持仓来源仍是复盘API）
  - 不新增API端点
  - 不改报警系统逻辑
  - 不改计划的编辑/删除流程

## 3. 数据模型

### 3.1 分组逻辑

```
今日计划数据（昨日计划 + 今日计划合并）
         │
         ├── 跨引用：提取计划项 stock 中的代码 → 判断是否在持仓中
         │
         ├── ✅ 是持仓 → 归入「🔴 持仓操作」
         │    显示：stock + 计划方向(买/卖/观察) + 条件 + 止损价 + 现价
         │
         └── ❌ 非持仓 → 归入「🟢 候选」
              显示：stock + 条件 + 止损
              按 buy/sell/watch 排序不分段
```

### 3.2 代码提取函数

```typescript
function extractCode(stock: string): string {
  // "华工科技(000988)" → "000988"
  const m = stock.match(/\((\d{6})\)/)
  return m ? m[1] : stock
}
```

## 4. 系统设计

### 4.1 前端组件结构

```
PlanLayer (collapsible)
├── 标题: 📋 今日计划 (N+M项)
│
├── 🔴 持仓操作（N只）
│   ├── [持仓+有计划的] 股票名 → 方向：条件  止损XX  现价XX
│   └── [持仓但无计划的] 股票名  止损XX  现价XX
│
├── 🟢 候选（M只）
│   └── [非持仓的计划项] 股票名  条件  止损  🔔
│
└── 🟡 计划报警（保持现有逻辑）
```

### 4.2 API 变化

无变化。PlanLayer 依然从这三个接口获取数据：
- `fetchWorkbenchPlan()` — 昨日/今日计划
- `fetchReviewToday()` — 持仓数据（含止损价）
- `fetchActiveAlarms()` — 报警数据

### 4.3 去重逻辑

```typescript
// step 1: 提取计划项中的股票代码
// step 2: 用 holdingCodes 判断每个计划项归属
// step 3: 持仓股 → 归入 holdingsGroup
// step 4: 非持仓 → 归入 candidateGroup
// step 5: holdingsGroup 同时合并该股的计划方向(买/卖/观察)
```
