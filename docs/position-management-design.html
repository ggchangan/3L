二、需求摘要

  1. 个股管理 — 新增、编辑、删除持仓个股
  2. 个股展示 — 以卡片风格展示每只持仓，显示名称、代码、方向、仓位%、止损价（含自动计算跌幅%）及当前股价/涨跌幅
  3. 新增持仓弹窗 — 搜索股票 → 显示当前价 → 选择方向 → 输入仓位%和止损价 → 自动计算止损跌幅%
  4. 方向分布饼图 — 复用现有 SVG 饼图实现，展示各方向仓位占比
  5. 持仓评估 — 基于市场环境评估当前持仓结构
  6. 持仓建议 — 基于评估结果给出可操作的调仓建议
  7. 不追溯历史 — 只维护当前快照，不记录仓位变更历史
  8. 盘后维护 — 当前版本只做盘后维护，盘中实时暂不介入
  9. 页面移出 private/ — holdings.html 作为正常前端页面，由构建系统处理
  10. 部署就绪 — 用户数据与代码分离，一键部署时清空持仓

三、部署架构（与 private/ 分离）

3.1 核心原则：代码与用户数据分离

部署给他人时，需确保不含任何持仓数据。关键在于：

  • 页面代码 — Holdings.html 从 private/ 移到 frontend/holdings.html，走标准构建流程
  • 用户数据 — holdings.json 仍存 DATA_DIR/private/，首次部署自动创建空持仓
  • 仓库跟踪 — .gitignore 忽略 data/3l/private/（作者的数据不推入仓库）

3.2 目录结构变化

当前改后
private/holdings.html（页面+脚本混合）frontend/holdings.html + frontend/css/pages/holdings.css
private/holdings.json（数据+页面混合目录）data/3l/private/holdings.json（仅数据）
private/trades.json不变，仍在 private/

3.3 首次部署行为

  • 如果 holdings.json 不存在 → 自动创建 {"holdings": [], "cash_ratio": 100}
  • 页面访问时如果持仓为空 → 显示 "暂无持仓" + 引导新增
  • 无需额外初始化脚本

3.4 路径定义

  • URL: /holdings.html
  • 不加入导航栏 — 私密页面，仅通过书签/直接URL访问
  • API 路径不变：/api/holdings（私密路径，数据不在前端HTML中硬编码）

五、数据模型

5.1 holdings.json（完整字段）

{
  "update_date": "2026-05-25",
  "holdings": [
    {
      "name": "药明康德",
      "code": "603259",
      "ratio": 14.70,
      "direction": "创新药",
      "stop_loss_price": 48.50,
      "sector": "化学制药",
      "structure": "上涨趋势",
      "stage": "上行",
      "price": 51.20,
      "change": 2.3,
      "change_pct": 4.71,
      "stop_loss_pct": -3.2
    }
  ],
  "cash_ratio": 9.16
}

字段说明：

字段类型说明
namestring股票名称
codestring股票代码（6位）
rationumber仓位占比百分比
directionstring所属方向（如"算力"、"新能源"）
stop_loss_pricenumber | null止损价（元），未设时为null
sectorstring同花顺行业分类（后台自动叠加）
structurestringK线结构：上涨趋势/下降趋势/区间震荡（后台分析）
stagestring走势阶段：上行/加速/缩量整理/滞涨等（后台分析）
pricenumber | null实时股价（腾讯接口叠加）
changenumber | null涨跌额
change_pctnumber | null涨跌幅百分比
stop_loss_pctnumber | null止损跌幅%（自动计算：(止损价-当前价)/当前价×100%）
cash_rationumber现金仓位（自动计算或手动维护）
update_datestring最后更新日期（yyyy-MM-dd）

九、持仓评估与建议逻辑

9.1 持仓评估指标

① 总仓位 vs 市场建议仓位
市场建议 来自 review_data.json 的 market.position_pct
（如"七至八成"映射为 70-80%）
实际总仓位 = 100% - cash_ratio
判定：实际 > 80% → ⚠️偏高（偏积极）；实际 60-80% → ✅合理；实际 < 60% → ⚠️偏低（偏保守）
总仓位分段描述：页面直接显示文字描述而非原始百分比比较
② 单只集中度
单只仓位 > 10% → ⚠️单只集中
单只仓位 > 15% → 🔴严重集中
每只带完整分位比例显示
③ 方向集中度（全部显示，按仓位降序）
单方向总仓位 > 30% → ⚠️方向集中
单方向总仓位 > 40% → 🔴严重集中
所有方向都显示，不再截断只显示 >30% 的
每项带 🔴/⚠️/✅ 图标
④ 现金充裕度
现金 < 5% → ⚠️现金不足，可能无法灵活调仓
现金 > 30% → ⚠️现金过多，仓位利用率低

9.2 评估排序与显示规则

  • 按严重程度排序：🔴 高 → ⚠️ 中 → ✅ 低
  • 所有评估项全部显示，不截断
  • 方向分布：所有方向按仓位降序排列，全部展示
  • 每项带图标（🔴/⚠️/✅）
  • 显示完整的分位比例数值
  • 总仓位：显示分段描述文字（合理/偏低/偏积极）

9.3 持仓建议生成

评估发现对应建议
总仓位偏高"建议适当减仓至X%以内，降低系统风险"
总仓位偏低"可适当加仓，目前仓位偏低，市场环境支持"
单只 > 10%"XX仓位超过10%，若跌破止损需及时降仓"
方向集中 > 30%"XX方向X只占比过高，可聚焦1-2只核心标的"
现金偏少"现金不足5%，建议预留部分现金应对调仓"
方向过度分散"XX方向多只标的，建议精简持仓数量"
某方向偏少且为主线"XX方向仅有1只，主线确认时可适当加仓"

步骤 1：后端 — 扩展 holdings_service

文件改动
services/holdings_service.py新增 save_holdings() + get_holdings_with_prices()

save_holdings(data):

  1. 校验 data 含 holdings 列表（list）和 cash_ratio（number, 0-100）
  2. 防误覆盖：如果当前 holdings.json 有 >50 只但请求写入 ≤10 只，拒绝
  3. 更新 update_date 为当天
  4. 原子写入（先写临时文件再 rename）
  5. 返回 {"success": true, "count": N}

get_holdings_with_prices():

  1. 读取 holdings.json
  2. 批量调用腾讯实时行情接口获取各股当前价
  3. 为每只持仓叠加 price、change、change_pct、stop_loss_pct
  4. 调用 industry_map 获取板块分类
  5. K线结构/阶段分析（上涨趋势/下降趋势/区间震荡 + 上行/加速/缩量整理/滞涨）
  6. 如果某只获取失败，price 和 change 用 null

  3. 处理异常（JSON 解析失败、校验失败等）

十一、备注与边界情况

  • 防误覆盖：POST保存时，如果当前已有 >50 只但写入 <10 只，拒绝写入
  • 止损价非必填：新增时可以不填止损价，此时 stop_loss_price 为 null
  • 方向下拉：从 direction_service 读取已启用的方向列表（编辑弹窗显示所有含禁用）
  • 搜索接口：复用现有三层搜索（全量A股名→板块映射→板块名兜底）
  • 实时行情：GET /api/holdings 调用腾讯接口，每次请求实时拉取
  • 私有页面：holdings.html 不加入导航栏，仅通过私密URL访问
  • 首次部署空数据：holdings.json 不存在时，get_holdings_with_prices 自动返回空持仓
  • 仓库不跟踪用户数据：.gitignore 已忽略 data/3l/private/，持仓数据不会误提交

十二、开发工作流（团队惯例）

本项目采用的开发流程，已在本模块开发中实践验证：

  1. 先设计 — 产出设计文档（含 wireframe + 数据模型 + API 设计 + 执行计划）
  2. 实现前后端 — 遵循 TDD 节奏：
    • 先写测试（RED）
    • 改代码使测试通过（GREEN）
    • 全回归测试确认不影响已有功能
  3. 确认前端效果 — 页面截图提交 / 用户直接查看并提反馈意见
  4. 确认后端正确 — API 测试验证 / 用户确认数据正确
  5. 迭代 — 用户反馈 → 修改 → 再确认（循环直至通过）
  6. 收尾 — 更新设计文档 → 导出 PDF → git commit → 推送到远端
本次迭代轮回统计：
· 设计文档版本：v2.0 → v2.1 → v2.2

· 后端修复轮次：约 3 轮（涨跌幅索引 / update_date / 方向接口路径）
· 总提交次数：10 次（含 1 次文档更新）
