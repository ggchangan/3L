# 数据更新时序拆分 — 设计文档 v1

> 状态: 设计阶段 | 日期: 2026-07-02

## 0. 问题总览

**现状：** 所有数据统一在 **交易日次日 06:00** 一次更新完成。

**痛点：** 用户当天下午/晚上无法复盘——个股K线（Tushare 17:00-18:00 可用）明明已经就绪，但因为板块K线（同花顺 T+1 延迟）必须等次日，所有数据都被拖到 06:00。

**目标：** 将更新管线拆分为两段——**当天傍晚先更新可用的数据让用户能做复盘，次日早上补齐板块K线**。

## 1. 背景与问题

### 1.1 各数据源可用时间

| 数据 | 数据源 | 可用时间 | 当前是否阻塞 |
|:----|:------|:--------|:----------:|
| 个股K线 | Tushare `api.daily` → `stock_daily` DB | **当天 17:00-18:00** | ✅ 可用但不更新 |
| 指数K线 | Tushare `api.index_daily` → `index_daily` DB | **当天 17:00-18:00** | ✅ 可用但不更新 |
| 行业映射 | `ths_index` + `ths_member` DB | **随时可用** | ✅ 可用但不更新 |
| 概念映射 | `ths_index` + `ths_member` DB | **随时可用** | ✅ 可用但不更新 |
| 板块实时快照(push2test) | push2test HTTP f3 | **收盘后立即可用** | ✅ 可用但不更新 |
| **板块K线 OHLC**（同花顺） | THS index API | **次日 06:00** | ❌ 这就是卡点 |

### 1.2 复盘页面实际依赖

复盘页面主要依赖：
- **个股K线** → 买点扫描、持仓盈亏 ✅ 当天可用
- **指数K线** → 大盘位置判断 ✅ 当天可用
- **板块实时涨跌幅** → 今日板块表现 ✅ 当天可用（push2test）
- **主线排名（20日涨幅）** → 当前基于同花顺板块K线OHLC计算 ⚠️ 次日才有
- **板块OHLC** → 板块K线图、技术指标 ⚠️ 次日才有

### 1.3 当前管线

```
06:00 交易日  ──►  Tushare增量 ──►  个股更新 ──►  指数更新 ──►  板块更新 ──►  清除缓存
                 (stock_daily)     (all_stocks)  (index_data)   (ths OHLC)
```

### 1.4 影响范围

`sector_daily.json` 消费者（受时序拆分影响）：
- `review_service.py` — 检查板块时效性、读取 leader 数据
- `market_health_service.py` — 市场健康度
- `concept_wave.py` — 概念波谷追踪
- `panic_monitor_service.py` — 恐慌监测
- `strong_trend_service.py` — 强势股筛选
- `data_layer.get_sector_daily()` — 统一读入口

## 2. 设计思想

### 2.1 核心理念

**能当天更新的数据不要拖到次日。**

个股K线、指数、行业/概念映射、板块实时快照——这些数据在收盘后 1-2 小时内完全可用。唯一不能当天更新的就是同花顺板块K线OHLC（T+1 延迟）。

### 2.2 方案选择

| 方案 | 优点 | 缺点 | 结论 |
|:----|:----|:----|:----:|
| **A: 拆两段** — 18:00 个股/指数/复盘，06:00 补板块K线 | 用户当天可复盘；改动小；风险低 | 主线排名次日才精准（但18:00可用近似值） | ✅ |
| B: 换数据源让板块K线当天可用 | 无 | 所有替代数据源（akshare/push2test）都没有完整板块OHLC历史 | ❌ |
| C: 保持现状 | 简单 | 用户当天无法复盘 | ❌ |

### 2.3 设计原则

1. **先出数据、后补精度** — 当天 18:00 用已有数据出复盘，次日 06:00 补精准板块K线
2. **不降低次日数据的质量** — 06:00 的板块K线依然走同花顺，和现在一样
3. **最小变更** — 不改数据模型，不改 API 接口，只改 `update_stock_data.py` 的 main() 执行顺序 + crontab

### 2.4 这个功能做什么/不做什么

**做什么：**
- 将 `update_stock_data.py` 拆分为可独立调用的子管线
- 新增 18:00 cron 执行「个股/指数/映射/复盘」管线
- 06:00 cron 减少为只做「板块K线 + 主线重算」
- 复盘页面在 18:00 后即可查看（板块数据用前一天的 OHLC + 当天 push2test）

**不做什么：**
- 不改复盘 API 返回格式（前端无感知）
- 不换板块K线数据源（同花顺依然是唯一可信源）
- 不做主线排名在当天 18:00 的近似计算（保持 06:00 的精准数据为权威版本）
- 不修改现有测试

## 3. 数据流

### 3.1 新的两段管线

```
18:00 交易日  ──►  管线A（Phase1）：
                     Tushare增量 ──► 个股更新 ──► 指数更新 ──► 行业/概念映射
                     ──► 扫个股卡片 ──► 复盘数据就绪
                     （跳过板块K线更新）
                     
06:00 次日    ──►  管线B（Phase2）：
                     板块K线更新（同花顺 THS OHLC）
                     ──► 主线重算 ──► 清除缓存
```

### 3.2 复盘页面数据时效性

| 数据 | 18:00 后看到 | 06:00 后补充 |
|:----|:------------|:------------|
| 个股K线 | ✅ 当天最新 | — |
| 持仓盈亏 | ✅ 当天最新 | — |
| 买点信号 | ✅ 当天最新 | — |
| 指数/大盘位置 | ✅ 当天最新 | — |
| 板块排名（今日涨跌） | ✅ push2test实时数据 | — |
| **主线排名（20日涨幅）** | ⚠️ 昨日THS + 今日push2test 推算 | ✅ 精准THS OHLC |
| **板块K线图** | ⚠️ 截至昨日 | ✅ 更新到当天 |
| **板块技术指标** | ⚠️ 截至昨日 | ✅ 更新到当天 |

## 4. 系统设计

### 4.1 执行管线拆分

`update_stock_data.py` 新增两个入口函数：

```python
def main_phase1():
    """18:00 管线 — 个股/指数/复盘"""
    _fetch_tushare_daily_incremental()  # Tushare增量
    update_industry_map()
    update_concept_maps()
    update_stocks()
    update_index()
    # 跳过 update_sectors()

def main_phase2():
    """06:00 管线 — 板块K线"""
    update_sectors()
    # 清除主线缓存（06:00 后主线数据用精确OHLC重算）
    _clear_mainline_cache()
```

### 4.2 crontab 变更

```diff
-0 6 * * 1-5  ... update_stock_data
+0 18 * * 1-5 ... python3 -m backend.core.update_stock_data --phase1
+0 6 * * 1-5  ... python3 -m backend.core.update_stock_data --phase2
```

### 4.3 复盘时效性标记（可选）

在复盘响应中增加 `data_freshness` 字段，告知用户板块数据时效：

```json
{
  "data_freshness": {

  }
}
```

## 6. 附录

### 6.1 替代方案 — 主线近似计算

考虑过在 phase1 中用 push2test 当日涨跌幅 + 昨日THS K线推算 20日涨幅近似值，精度分析：

```
推算方法：
  近似今日收盘 = 昨日收盘 × (1 + push2test.change_pct / 100)
  近似20日涨幅 = 近似今日收盘 / 20日前收盘 - 1

误差来源：
  1. push2test 的 change_pct 是精确值，精度高 ✅
  2. 即使板块K线OHLC有误差，对排名影响很小（板块间差距通常 > 0.5%）
  
结论：可做，约20行代码，风险低。
```

暂不实现，先观察用户是否需要这个近似值再定。
