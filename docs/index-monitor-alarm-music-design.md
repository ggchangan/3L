# 大盘指数监测 + 报警音乐系统 — 设计文档 v0.4

> 更新 2026-05-30：报警系统 v3.5 已完成。本节标记为"已实现"。
> 时间回溯回放功能仍为草案，暂缓。

## 0. 设计原则（从产品角度出发）

### 用户画像

一个A股交易者，盘中开着盯盘页面做自己的事，需要：
- **重要的事立刻知道** — 止损触发必须响 + 推手机
- **不重要的别吵** — 偏差/大盘预警不用推微信，音乐播一下即可
- **自己控制开关** — 点按钮才播音乐，不是页面一打开就响
- **反馈明确** — 点了开关立刻知道生效了
- **一个接一个** — 别多个同时响

### 角色分工

| 角色 | 职责 |
|------|------|
| **产品经理** | 定义什么报警"值得响"、什么推微信、什么忽略 |
| **前端** | 按钮交互、单音频元素播放、toast展示、不要跟浏览器策略较劲 |
| **后端** | 30秒检测条件 → 写`alarms.json` → 按规则推WxPusher |

### 前端音频引擎（重新设计，吸取教训）

**核心原则：一个 `<audio>` 元素就够了。**

```tsx
// 不要多个audio元素，不要JSX控制muted，不要模块级变量
const audioRef = useRef<HTMLAudioElement>(null)

// 点按钮开 → unmute + 备用音效
const toggleSound = () => {
  if (soundEnabled) return // 关→开
  const el = audioRef.current!
  el.muted = false
  el.volume = 1
  // 立刻播500ms静音音效让用户感知"开了"
  playShortBeep(el)
}

// 报警来 → 换src → play（按钮已点，浏览器不拦）
const playAlarm = (type: AlarmType) => {
  const el = audioRef.current!
  el.src = soundConfig[type].url
  el.play() // 用户已经点过按钮，Chrome放行
}
```

**关键设计决策（封死之前的坑）：**
1. 不用JSX `muted` 属性，用 `useRef` 直接操作DOM，React重渲染不影响
2. 不用模块级 `_soundEnabled` 变量，全部用组件state
3. 按钮点击本身就是用户手势，浏览器不会拦截后续 `play()` 调用
4. 每页面session从静音开始，刷新重来（不存localStorage）
5. 开声音时扫当前toast列表，补播第一个

### 微信推送规则（产品决定）

| 报警类型 | 前端响音乐 | 推微信 |
|---------|:---------:|:-----:|
| **stop** / 止损触发 | ✅ 必须响 | ✅ 必须推 |
| **price** / 价格报警 | ✅ 响 | ✅ 推 |
| **deviation** / 偏离 | ✅ 响 | ❌ 不推（太频繁） |
| **market** / 大盘预警 | ✅ 响 | ❌ 不推 |
| **market_critical** / 系统风险 | ✅ 响 | ✅ 推 |
| info / abnormal / 其他 | ❌ 不播 | ❌ 不推 |

## 1. 目标

**核心痛点**：2026-05-29 科创50单日跌5%+跌破支撑位，导致大亏。需要自动监测
关键指数风险，提前/实时预警。

**理念**：报警不再是短促"哔"声，而是**按等级播不同歌曲/旋律**，让人一听就知道

### 数据刷新（与复盘架构一致）

**遵循"cron 只拉原始数据，不计算"的原则：**

- **每日 17:00 cron**：拉/刷新三只指数的 K 线数据（一次性拉 60~200 天），写入 `index_data.json`
- **盘中（盯盘）**：`check_alerts` 只**读取**已存数据（ema60、昨收价），通过腾讯行情 API 拉实时价做判断
- **不**在盘中拉历史 K 线

### 初始建库

首次执行一次手动脚本：
1. 拉上证指数 200 天 K 线（已有 `index_sh_data.json`，迁移即可）
2. 拉科创50 60 天 K 线（新拉）
3. 拉中证全指 60 天 K 线（已有拉数函数）
4. 计算各指数 EMA60
5. 统一写入 `index_data.json`

## 3. 报警等级体系

### 五级结构

| 等级 | 标识 | 触发条件 | 音乐类型 | 示例场景 |
|------|------|---------|---------|---------|
| 🔴 **止损** | `stop` | 持仓跌破止损价 | 紧张/高亢旋律 | 持仓股止损触发 |
| 🟡 **异动** | `deviation` | 核心股涨跌 > 阈值(6%) | 中等节奏旋律 | 核心股单日异动 |
| 🟠 **大盘** | `market` | 任一指数跌 > 3% | 低沉/深沉旋律 | 指数大跌 |
| 🟠 **破位** | `market` | 任一指数跌破 EMA60 | 同大盘旋律（但标记"破位"） | 趋势性破位 |
| 🔴 **系统风险** | `market_critical` | 跌>3% 且 破 EMA60 (双重确认) | 同 stop 最紧急旋律 | 最严重的系统性风险 |

### 报警对象

| 对象 | 来源 | 已有/新建 |
|------|------|---------|
| 持仓止损 | `alarms.json` (holdings_auto) | ✅ 已有 |
| 用户手动报警 | `alarms.json` (manual) | ✅ 已有 |
| 核心股自动偏差 | `check_alerts.py` (core_stocks) | ✅ 已有 |
| **指数监测** | `check_alerts.py` (**新增**) | 🆕 新建 |

## 4. 后端逻辑

### `check_alerts.py` 新增指数检查（与个股报警同一 30 秒轮询）

在 `check_all_alerts()` 末尾追加：

```
for 每只指数 in [上证, 科创50, 中证全指]:
  1. 从 index_data.json 读最近数据 (昨收价, EMA60)
  2. 从腾讯行情 API 拉实时价 (同个股报警接口)
  3. 算当日涨跌幅 = (实时价 - 昨收) / 昨收
  4. 判定:
     - 跌 > 3% → 触发 market 报警
     - 实时价 < EMA60 × 0.98 → 触发 market 报警 (破位)
     - 两个都满足 → 触发 market_critical (最高级)
  5. 写入 triggered[] 列表，8小时内不重复触发同条件
```

### 报警去重

- 每只指数每条件 8 小时内只触发一次（防刷屏）
- 用 `triggered_at` + 指数代码 + 条件类型做 key

## 5. 报警音乐系统

### 5.1 配置存储 + 微信通知

```json
// data/public/alarm_sounds.json
{
  "stop":    { "url": "/assets/sounds/stop.wav",    "name": "止损音效" },
  "stock":   { "url": "/assets/sounds/stock.wav",   "name": "个股音效" },
  "market":  { "url": "/assets/sounds/market.wav",  "name": "大盘音效" },
  "market_critical": { "url": "/assets/sounds/stop.wav", "name": "同止损音效(最高级)" }
}
```

### 5.2 微信通知（WxPusher，v3.5.0+）

报警触发时，后端通过 **WxPusher** HTTP API 直接推送微信，不再依赖 Hermes agent 或 cron job：

```python
# wxpush_sender.py
requests.post("https://wxpusher.zjiecode.com/api/send/message", json={
    "appToken": APP_TOKEN,
    "content": markdown_text,
    "contentType": 3,
    "uids": [UID]
})
```

`.env` 配置：
- `WXPUSHER_TOKEN=AT_xxx` — WxPusher 应用 Token
- `WXPUSHER_UID=UID_xxx` — 接收用户 UID（可在线配置页修改）

### 默认旋律

| 文件 | 时长 | 调性 | 描述 |
|------|------|------|------|
| `stop.wav` | 15秒 | 880Hz↔440Hz交替 | 快速警笛，紧张 |
| `stock.wav` | 15秒 | C5→D5→E5→F5→G5上行 | 中等速度音阶 |
| `market.wav` | 15秒 | 110Hz↔165Hz缓慢交替 | 低沉缓慢，庄重 |

用户可替换为任意 MP3（3分钟的歌也行）。

### API

```
GET /api/alarm-sounds → { alarms: { stop: {url, name}, ... } }
```

### 前端播放策略

- 每个报警类型对应一个隐藏 `<audio>` 元素
- 页面首次用户点击时预加载所有音效（规避 autoplay 策略）
- `<audio>` 的 `ended` 事件监听播放完毕，播下一首（排队）
- 同一类型重复触发 → 不打断不重播
- 不同类型排队 → 先播完当前再播下一个
- 用户关 toast → 停止该歌

## 9. 时间回溯回放功能（Time-machine Replay）— 设计草案

> 需求：拉取完整交易日分钟级数据，倍速重演全天交易过程，验证报警触发、买卖点信号、声音播放是否符合预期。

### 9.1 用户场景

- **晨会复盘**：用2x/5x回放昨天盘面，看止损触发时声音/弹窗是否正确
- **新规则验证**：改了报警条件后，回放历史交易日验证是否误报/漏报
- **教学/分析**：慢速回放关键转折点（如科创50破EMA60的瞬间），看信号联动

### 9.2 数据源

| 数据类型 | 精度 | 数据量 | 来源 |
|---------|:----:|:-----:|------|
| 大盘指数（上证/科创50/中证全指） | 分钟级 | ~240条/天/指数 | 腾讯 ifzq 或 ak share |
| 个股（持仓+核心股） | 分钟级 | ~240条/天/股 | 腾讯个股分钟K线 |
| 板块指数 | 分钟级 | ~240条/天/板块 | ak share 板块分钟线 |

**拉取策略**：
- 首次触发回放时按需拉取（不提前拉，不浪费存储）
- 拉完缓存到 `data/replay/YYYYMMDD/` 目录
- 同一天只拉一次

### 9.3 回放引擎（后端）

```
GET /api/replay/start?date=20260529&speed=5&indices=000001,000688,000985&stocks=002371,002436
```

返回 `replay_id` + 时间片总数。

```
POST /api/replay/step?replay_id=xxx&tick=N
```

返回该分钟的所有数据：
- 各指数实时价、涨跌幅、EMA位置
- 各个股实时价、涨跌幅、均线位置
- 符合触发条件的报警列表（`{type, msg, alarmId}`）
- 买卖点信号（`{action, stock, price, reason}`）

### 9.4 前端实现

**进入回放模式**：
- 盯盘页面添加一个"⏪ 回放"按钮
- 点后弹出配置面板：选日期、速度(1x/2x/5x/10x)、监测范围
- 开始后整个盯盘页面进入"回放模式"

**回放模式界面**：
- 顶部显示进度条（当前时间片/总时间片）+ 播放/暂停/调速控件
- 页面所有数据面板**实时更新**——大盘指数曲线、个股行情、板块排行都按倍速推进
- 底部原有报警图层正常工作：到触发时间点就弹toast、播音乐、推微信（可选项）
- 声音按钮同样工作：点开听到回放中的报警音乐

**退出回放**：
- 回到实时模式，一切数据恢复正常轮询

### 9.5 关键设计决策

1. **复用现有数据面板** — 不新建回放专用组件，用同一个Monitor页面但切换数据源（实时 vs 回放）
2. **后端逐帧喂数据** — 前端不存全量数据，后端每步返回当前帧的全部状态，前端渲染
3. **报警引擎复用** — 后端 `check_alerts` 逻辑可直接对接回放数据（换入历史价，跑同一套判定）
4. **速度控制** — 用`setInterval`在倍速下推进，5x = 每12秒推进1分钟

### 9.6 实施步骤（初步）

```
Step 1: 后端 - 分钟级数据拉取API + 缓存设计
Step 2: 后端 - 回放引擎（逐帧喂数据 + 报警判定）
Step 3: 前端 - 回放入口 + 配置面板 + 进度控制
Step 4: 前端 - 数据面板接入回放数据（替代实时轮询）
Step 5: 集成测试 - 选定某天回放全流程
```
