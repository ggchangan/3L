📦 3L 前端架构迁移方案

版本：v1.0 | 日期：2026-05-26 | 状态：草案

1. 现状与问题

当前架构

frontend/
├── *.html              # 15 个静态 HTML 页面
├── public/js/pages/    # 15 个对应的 vanilla JS 文件
├── css/pages/          # 15 个对应的 CSS 文件
├── dist/               # Vite 构建产物（CSS 压缩 + 前缀）
├── build.py            # 构建脚本
├── vite.config.js      # Vite 配置（多页模式，仅处理 CSS）
└── package.json        # 仅 vite + pinyin-pro

核心问题：

#问题影响
1数据变更后页面不会自动重绘用户改趋势列表 → 页面不变，需要手动刷新/重新生成
2JS 全局作用域，无组件化CSS 类名冲突需手动排查（stock_card.js 的 chart-container 被 monitor.css 覆盖过）
3状态管理靠 DOM + 全局变量一个页面改数据，另一个页面不知道
415 个 HTML + 15 个 JS + 15 个 CSS 独立文件公共逻辑复制粘贴（stock_card.js、nav.js 每个页面单独引用）

架构决策

决策选择理由
框架React + TypeScript用户需求：数据变 → 组件自动重绘
构建Vite（已有，升级配置）Vite 已在用，加 React 插件即可
路由React Router（单页应用）多页模式下组件无法共享状态
样式CSS Modules避免全局类名冲突，逐个迁移
状态管理React Context + useApi hooks目前不需要 Redux，数据量小
图表SVG React 组件后端生成的 SVG 直接内嵌，无需额外库

3. 迁移策略：渐进式替换

核心原则：一次迁一页，两边共存

新（React）和旧（静态 HTML）同时在线，不走 big bang。

后端 server.py:
  /monitor          →  React 版 monitor.html  新
  /review           →  旧 review.html         逐步替换
  /watchlist        →  旧 watchlist.html      逐步替换
  ...

迁移顺序

阶段页面原因复杂度
Phase 1监控(monitor) + 个股卡片(stock_card)复用最广，组件化收益最大⭐⭐
Phase 2复盘(review) + 工作台(journal)核心流程页，数据流复杂⭐⭐⭐
Phase 3自选股(watchlist) + 趋势(trend_candidates)交互复杂（搜索/排序/勾选）⭐⭐⭐
Phase 4其余页面（涨幅/宏观/行业/tips/模拟/技巧）数据展示为主，迁移快⭐

4. 数据流设计

React 组件
  ↓
useApi hook
  ↓
fetch('/api/...')    ← 后端 REST API 完全不变
  ↓
JSON response
  ↓
组件 setState → 自动重绘 ✅

关键：后端 API 一条不改。 所有改进在前端层面。

当前所有 API 端点：

API用途迁移后
/api/review/get复盘数据React 组件 fetch
/api/review/dates复盘日期列表React 组件 fetch

/api/trend/candidates趋势候选React 组件 fetch
/api/trend/tracked已跟踪趋势React 组件 fetch

/api/monitor/...盯盘数据React 组件 fetch
/api/stock/...个股分析React 组件 fetch

5. 构建与部署

项目说明
开发npm run dev → Vite dev server (port 5173) → proxy API 到 8080
构建npm run build → Vite 编译 TypeScript + 打包 → dist/
部署python3 build.py → 复制 dist/ 到前端目录 → systemctl restart 3l-server
共存旧 HTML 文件仍保留，server.py 通过路由前缀分发新旧版本

build.py 变更： 简化。不再需要 inject_stock_card_css 等补丁，因为 React 的 CSS Modules 自带作用域隔离。

6. 风险与注意事项

  1. 后端不动 — 这是规则。迁移期间不改后端 API，确保新旧前端都能用
  2. 不重写逻辑 — 从现有 JS 直接翻译成 React，不做功能改进。改进是另一个话题
  3. SVG 图表 — 后端生成的 SVG 用 dangerouslySetInnerHTML 内嵌，不需要重写
  4. 旧页面保留 — 出问题时快速回退：改 server.py 路由指向旧 HTML 即可
  5. TypeScript 严格度 — 先 loose（strict: false），稳定后再收严

7. 执行计划

Phase 0.5：清理冗余文件（1 次提交）

React 迁移前先把项目根的无用旧文件清理掉，避免混淆。

步骤 1: 确认哪些文件是活的
  server.py 的 FE_DIR = frontend/dist/（优先）
  回退路径 = 项目根目录（只有 dist 里没有时才用）

步骤 2: 检查项目根 14 个 .html 是否被任何地方引用
  grep -r "\.html" server.py backend/ 确认引用关系

步骤 3: 删除确认无用的旧文件
  项目根目录的 *.html（确认无引用后）
  项目根目录的 CSS/JS（如果存在）
  旧 build.py 中的 inject_stock_card_css 等补丁（迁移后不再需要）

步骤 4: 验证
  systemctl restart 3l-server → 页面正常打开
  所有 API 端点正常返回

测试:
  手动打开 monitor/review/workbench 等页面，确认正常渲染
  pytest tests/ 全回归

Phase 0：基建搭建（1 次提交）

步骤 1: 安装依赖
  npm install react react-dom react-router-dom
  npm install -D @types/react @types/react-dom @vitejs/plugin-react typescript

步骤 2: Vite 配置 + TypeScript 配置
  vite.config.ts → 加入 @vitejs/plugin-react
  tsconfig.json  → 基础配置（strict: false）
  src/main.tsx   → React 入口
  src/App.tsx    → 路由定义

步骤 3: 验证
  npm run dev → 能打开 React 空页面
  npm run build → 构建成功
  python3 build.py → 部署正常

测试:
  tests/test_frontend_build.py → 验证构建不报错
  手动打开浏览器确认首页渲染

Phase 1：Monitor 页面 + StockCard 组件（首期迁移）

步骤 1: 类型定义
  src/types/index.ts → API 返回数据接口
  
步骤 2: API 封装

测试:
  tests/test_monitor_page.py → API 响应结构验证
  手动对比新旧两个版本的数据一致性

...后续 Phase 2-4 逐步推进，每页独立提交。
