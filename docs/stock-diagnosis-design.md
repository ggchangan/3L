# 3L 个股诊断系统 — 设计文档

> 版本: v0.1 | 日期: 2026-05-29 | 分支: feature/stock-analysis-enhance

---

## 一、背景

当前个股分析页面（主 SPA `/stock_analysis` + 独立页面 `/stock-analysis`）只展示技术面数据（结构/阶段/均线/信号/买点），缺乏从**财务面**和**风险面**对个股质量的综合评估。

用户输入个股后，除了看到技术分析，还想知道：
1. 这只股票财务质量怎么样？（ROE、增长、负债）
2. 有没有什么风险？（高负债、业绩下滑、趋势走坏）
3. 综合来看这股票得多少分？（直观的评分+等级）

---

## 二、功能定义

### 2.1 核心能力

在个股分析页面，用户搜索一只股票后，系统自动完成**三维度诊断**：

```
输入个股 → 并行获取：
  ├── 趋势面（已有 3L 分析，直接复用）
  ├── 财务面（akshare 财务分析指标，新加）
  └── 风险面（综合判定，新加）
       ↓
输出：总分 + 等级(A/B/C/D) + 各维度详情 + 说明文字
```

### 2.2 过渡方案（新闻占位）

新闻/消息面情感分析本次不做 **完整实现**，但在诊断结果中预留占位：

```json
"news": {
  "status": "coming_soon",
  "message": "消息面分析即将上线"
}
```

前端展示一个「📰 消息面 — 即将上线」的灰色占位卡片，视觉上保持四列布局对称，后续接入数据源后直接替换。

### 2.3 非目标（本次不做）

- ❌ 批量扫描全部自选股（未来可加）
- ❌ 新闻/消息面情感分析（akshare 新闻接口慢且不稳定，等找稳定源再加）
- ❌ 历史诊断对比
- ❌ 诊断报告导出

## 三、数据模型

### 3.1 API 响应结构

在现有 `/api/stock-analysis` 响应中新增 `diagnosis` 字段：

```json
{
  // ... 现有字段（code, name, structure, stage, signal 等）
  "diagnosis": {
    "total_score": 78,
    "grade": "B",
    "detail": {
      "financial": {
        "score": 32,
        "max": 40,
        "data": {
          "roe": 16.14,
          "eps": 16.83,
          "revenue_growth": 17.04,
          "profit_growth": 42.18,
          "debt_ratio": 62.32,
          "current_ratio": 1.60,
          "quick_ratio": 1.34,
          "net_margin": 17.61
        },
        "remarks": ["ROE 16.1% 良好", "净利润增长 42.2% 高速"]
      },
      "trend": {
        "score": 28,
        "max": 40,
        "remarks": ["上升趋势 +8", "阶段上行 +5", "买入信号 +5"]
      },
      "risk": {
        "score": 18,
        "max": 20,
        "level": "低风险",
        "items": ["资产负债率 62.3% 适中"]
      }
    },
    "cost_ms": 520
  }
}
```

### 3.2 评分规则

| 维度 | 满分 | 评分依据 |
|:-----|:----:|:---------|
| **财务面** | 40 | ROE(10)、净利润增长率(8)、营收增长率(5)、资产负债率(5)、基础分(12) |
| **趋势面** | 40 | 结构(8)、阶段(5)、信号(5)、主线(5)、买点(5)、乖离率(-3)、基础分(15) |
| **风险面** | 20 | 扣分制：净利润下滑(-8)、高负债(-5)、流动性差(-5)、趋势评分低(-5) |
| **总分** | 100 | 财务 + 趋势 + 风险 |

| 等级 | 分数区间 | 含义 |
|:----|:---------|:-----|
| A | ≥85 | 优秀 — 基本面好+趋势强+风险低 |
| B | 70~84 | 良好 — 大多数维度健康 |
| C | 55~69 | 一般 — 有需要注意的问题 |
| D | <55 | 较差 — 多个维度预警 |

## 五、数据流

```
用户输入 "300750"
    │
    ▼
前端 fetch(/api/stock-analysis?q=300750)
    │
    ▼
server.py → stock.py → search_and_analyze()
    │                              │
    │                  ┌───────────┴───────────┐
    │                  │                       │
    │           趋势分析(已有)          诊断服务(新)
    │                                     │
    │                     ┌───────────────┴───────────────┐
    │                     │                               │
    │              财务面(akshare)                  风险判定
    │              0.5s缓存1小时                   纯计算<1ms
    │                     │                               │
    │                     └───────────────┬───────────────┘
    │                                     │
    │                             合并结果 + 评分
    │                                     │
    └─────────────────┬───────────────────┘
                      │
                      ▼
          返回 JSON（含 diagnosis 字段）
                      │
                      ▼
           前端渲染诊断区块
```

## 六、后端实现计划

### 6.1 新增文件

| 文件 | 说明 |
|:-----|:------|
| `server/backend/services/diagnosis_service.py` | 诊断服务：财务获取 + 评分 + 风险判定 |

### 6.2 修改文件

| 文件 | 改动 |
|:-----|:------|
| `server/backend/api/stock.py` | `search_and_analyze()` 后调用 `compute_diagnosis()`，合并到返回值 |
| `server/frontend/src/pages/StockAnalysis.tsx` | 诊断结果渲染组件 |
| `server/frontend/src/pages/StockAnalysis.css` | 诊断区块样式 |
| `analysis/frontend/index.html` | 独立页面的诊断渲染（JS + CSS） |

### 6.3 缓存策略

- 财务数据：内存缓存 1 小时（同一个股在 1 小时内重复查询不重复请求 akshare）
- 趋势面评分：纯计算，不缓存
- 风险面评分：纯计算，不缓存

## 八、注意事项

1. **akshare 首次请求慢**：首次请求 akshare 会下载数据文件（约 2-3s），缓存后仅 ~0.5s
2. **财务数据年份**：`start_year='2025'` 取最近财报，年报/季报按实际发布日期更新
3. **错误降级**：如果 akshare 请求失败，financial 维度返回 0 分并标记「数据获取失败」，不影响页面其他功能
4. **两个页面同步**：主 SPA 用 React 组件渲染，独立页面用纯 HTML+JS 渲染，逻辑保持一致
